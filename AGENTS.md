# AGENTS.md

Guidance for AI coding agents working in this repository.

## Project Purpose

This repository powers https://mgifford.github.io as a curated catalog of repositories owned by mgifford.

The site has two goals:

- Public portfolio view for curated, high-signal projects.
- Owner maintenance view for identifying unfinished work (open PRs, stale issues, notifications, README quality, archived state).

## Stack and Architecture

- Frontend: static HTML/CSS/JS
  - index.html
  - src/app.js
  - src/styles.css
- Data and automation: Node.js scripts
  - scripts/sync-repos.mjs
  - scripts/build-site-data.mjs
  - scripts/capture-pages-screenshots.mjs
- Curation: YAML
  - data/curation.yml
- Generated artifacts
  - data/generated/repos.json
  - data/generated/changes.json
  - data/generated/screenshots.json
  - data/site-data.json
  - reports/changes-latest.md
- Automation
  - .github/workflows/weekly-sync.yml
  - .github/workflows/monthly-screenshots.yml

## Core Principles

- Keep the site fully static and GitHub Pages compatible.
- Keep curation human-editable in YAML.
- Keep generated files deterministic and machine-written.
- Separate public signals from owner-only maintenance signals.
- Avoid adding server-side runtime dependencies.

## Agent Operating Rules

1. Respect manual curation.
- Do not overwrite data/curation.yml structure unless asked.
- Preserve manual overrides and featured lists.

2. Treat generated files as build output.
- Regenerate instead of hand-editing files under data/generated/ when possible.
- If scripts change output schema, update build-site-data.mjs and frontend usage together.

3. Keep owner mode safe.
- Never commit personal tokens, credentials, or session data.
- Owner mode token use must remain browser-local.

4. Keep workflows lightweight.
- Weekly job: repository sync and site-data rebuild.
- Monthly job: screenshot refresh and site-data rebuild.
- Avoid adding AI-dependent cron jobs unless explicitly requested.

5. Prefer incremental, readable changes.
- Keep scripts composable.
- Avoid broad rewrites when a small patch is enough.

## Common Commands

Install dependencies:

```bash
npm install
```

Run full sync pipeline:

```bash
npm run sync:all
```

Rebuild merged frontend data:

```bash
npm run build:data
```

Capture screenshots:

```bash
npm run screenshot:pages
```

Run local static server:

```bash
npm run start
```

## Commit and PR Conventions

Use clear, scoped messages that describe intent and impact.

Commit message style:

- Prefer conventional prefixes such as `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`.
- Keep subject line concise and action-oriented.
- Mention affected area when useful, for example: `feat(ui): add owner impact dashboard`.

Pull request title style:

- Use a short scope in square brackets followed by a plain-language summary.
- Format: `[scope] Short description`
- Examples:
  - `[catalog] Improve curated homepage defaults`
  - `[automation] Capture screenshots only for changed repos`
  - `[docs] Clarify token scopes for workflows`

PR checklist expectations:

- Run relevant validation commands before opening or merging.
- Update related docs when behavior or policy changes.
- Keep generated-file updates tied to script or pipeline changes.

## Data Contract Expectations

Each repo record consumed by the frontend should include:

- identity: name, fullName, url
- status: archived, fork, visibility
- metrics: stars, watchers, openIssues, updatedAt
- metadata: language, topics, description, homepage, hasPages
- enrichment: readme summary and quality signal
- curation overlay: theme, featured flag, optional screenshot, optional highlight
- presentation overrides when curated: cardTitle, cardSummary, featuredNarrative, manualSortRank, visibility

If this contract changes, update scripts and frontend together in the same change.

## Screenshot Policy

- Capture screenshots only for repos with a valid homepage URL or GitHub Pages indicator.
- Keep capture count bounded via MAX_SCREENSHOTS.
- Failures should be recorded in screenshots.json without failing the whole catalog unnecessarily.

## What To Avoid

- Reintroducing old reveal.js assets or build tooling.
- Committing secrets to the repository.
- Replacing curated summaries with low-quality autogenerated text without review.
- Converting this project into a server-rendered app unless explicitly requested.
